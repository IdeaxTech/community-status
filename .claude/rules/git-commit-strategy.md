# Git コミット戦略

すべてのフローでコミットするタイミングと方法の単一情報源。

## バリデーションゲート（/work — 標準フロー）

最後ではなく、各スライスが検証をパスした後にコミットする：

1. スライスを実装する（1つ以上の受け入れ基準）
2. `./scripts/run-verify.sh`（または同等）を実行する
3. 検証がパス → ファイルをステージして conventional format でコミット
4. 検証が失敗 → コミット前に修正する
5. 次のスライスに繰り返す

これにより個別に検証された変更のクリーンな履歴が作成される。

## セッション終了 / コンパクション前の WIP コミット

feature ブランチでセッションが終了するかコンテキストコンパクションが起きた場合：

- 未コミットの変更を `wip:` プレフィックスで自動的にコミットする
- feature ブランチのみ（main/master では決して行わない）
- メッセージ形式：`wip: checkpoint before <reason>`
- 後でスカッシュしても安全

## コミットメッセージの安全なクォート

ダブルクォートされた `git commit -m "..."` の中のシェルコマンド置換はシークレットを漏洩させる可能性がある。

**ルール**：

- **複数行メッセージ**：シングルクォートの区切りで HEREDOC を使う：
  ```sh
  git commit -m "$(cat <<'EOF'
  feat: add feature

  Details here.
  EOF
  )"
  ```
- **1行メッセージ**：シングルクォートを優先：`git commit -m 'fix: description'`
- **禁止**：ダブルクォートの `-m "..."` 引数の中にバッククォートや `$(...)` を置く

## コミット形式

Conventional Commits に従う：

```
<type>: <description>

<optional body>
```

タイプ：`feat`、`fix`、`refactor`、`docs`、`test`、`chore`、`perf`、`ci`、`wip`

`wip` タイプは自動チェックポイントのみに予約されている。
